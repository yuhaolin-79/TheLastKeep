# 游戏崩溃排查与稳定性修复总结

## 1. 问题现象

游戏运行时多次弹出 Qt Debug Error，主要信息如下：

- 程序位置：`Qt6Cored.dll`
- 文件位置：`qarraydataops.h:44`
- 断言内容：`this->isMutable() || b == e`
- 弹窗会重复出现，最后游戏无法继续运行。

第二关结束后也出现过闪退，所以本次同时检查了战斗更新、关卡结束和页面切换逻辑。

## 2. 排查结论

通过最新一次崩溃转储文件确认：

1. 崩溃发生在程序启动后约 6 秒。
2. 调用栈停在 `GamePage::setupUi()`。
3. 最后进入的位置是 `new QGraphicsView(m_scene, this)` 的构造过程。
4. 这一次崩溃不是第二关结束或结果页面切换直接触发的。

旧转储最后停在 Qt 内部代码中，无法只根据这一处断言确定唯一的项目代码原因。因此没有直接把问题归到某一行代码，而是继续检查了容器修改、对象销毁、帧更新和结束信号。

完整重新编译后，当前 MSVC Debug 版本已经无法复现这次启动崩溃。说明旧构建产物也可能参与了问题，但目前没有足够证据把它确定为唯一原因。

## 3. 已发现的风险

### 3.1 帧更新可能继续执行

敌人更新时可能已经触发城堡失败。原来的当前帧仍有机会继续执行子弹、碰撞和清理逻辑。

这会增加以下风险：

- 已结束的战斗继续修改对象。
- 同一帧重复进入结束判断。
- 已删除对象仍被后续逻辑访问。

### 3.2 结束信号可能重复处理

胜利或失败发生后，如果没有独立的结束锁，后续更新可能再次发送结束信号。页面切换、计时器和战斗清理也可能被重复执行。

### 3.3 容器清理过程不够稳定

原来的敌人清理逻辑会新建一个 `QList`，向其中追加存活敌人，最后再与原容器交换。

Qt 容器使用隐式共享。容器发生非只读操作时可能分离数据，修改容器也可能使现有迭代器失效。旧逻辑不一定每次都会崩溃，但会让对象引用和容器状态更难确认。

### 3.4 对象销毁顺序需要固定

子弹会保存敌人目标。清理战斗对象时，如果先删除敌人，再处理仍然指向敌人的子弹，就可能留下无效指针。

## 4. 本次修改

### 4.1 禁止帧更新重入

`GameController` 增加了帧更新状态：

```cpp
bool m_updatingFrame = false;
```

进入 `updateFrame()` 时先检查该状态。更新过程中使用 `QScopedValueRollback` 临时设置状态，函数正常结束、提前返回或异常退出时都会恢复原值。

### 4.2 胜利和失败只处理一次

`GameController` 增加了结束状态：

```cpp
bool m_finishEmitted = false;
```

结束流程现在按以下顺序执行：

1. 先设置结束状态。
2. 停止继续更新。
3. 修改游戏状态。
4. 发送胜利或失败信号。

这样可以避免同一局重复发送结束信号。

### 4.3 失败后立即停止当前帧

敌人更新完成后，如果游戏已经不是运行状态，当前帧不再继续执行子弹更新和碰撞检测，只进行必要的死亡敌人清理。

### 4.4 改为原容器倒序删除

死亡敌人现在直接在原容器中按索引倒序删除，不再创建存活敌人列表，也不再执行容器交换。

删除敌人前先移除以该敌人为目标的子弹，然后再删除敌人对象。

### 4.5 固定战斗对象清理顺序

战斗对象统一按以下顺序删除：

1. 子弹
2. 敌人
3. 防御塔

每次从容器末尾取出一个对象再删除，容器状态始终和剩余对象一致。

### 4.6 防止关卡结束重复切换

`GamePage` 的结束处理状态只在新关卡开始时重置。停止当前游戏时不再重置，避免清理过程中再次接收并处理结束结果。

结果页面的返回计时器为单次计时器。离开结果页面时会停止计时器，防止旧页面的超时信号影响新页面。

页面切换继续使用延迟到下一次事件循环的方式执行，使当前信号处理先正常结束，再清理游戏页面。

## 5. 当前结束流程

### 普通关卡胜利

- 引导关卡、第一关和第二关显示原有胜利结果和对应按钮。
- 不显示最终胜利背景图。
- 玩家通过按钮进入下一关或返回主界面。

### 最后一关胜利

- 显示胜利背景图。
- 不显示普通结果按钮。
- 10 秒后返回主界面。

### 任意关卡失败

- 显示失败背景图。
- 不显示普通结果按钮。
- 10 秒后返回主界面。

| 检查项目 | 结果 |

| 崩溃转储调用栈 | 已确认最后一次崩溃发生在启动阶段 |

| MSVC Debug 使用 Windows 平台插件启动 | 连续运行超过 10 秒，未出现相同断言和访问异常 |
| 启动页面构造顺序检查 | 未发现数组越界或未初始化成员 |
| 战斗结束和对象清理逻辑检查 | 已增加重入、重复结束和无效指针保护 |

目前完成的是编译、启动和代码路径检查。还没有自动化覆盖从第一波开始一直打到第三关结束的完整流程。后续仍需要实际游玩验证以下情况：

- 每一关胜利后只切换一次页面。
- 每一关失败后只显示一次失败页面。
- 最后一关胜利后 10 秒返回主界面。
- 结果页面停留期间不会继续生成敌人或更新战斗对象。
- 快速连续点击结果按钮不会重复进入下一关。

## 7. 后续代码要求

1. 遍历 Qt 容器时，不在同一个遍历过程中修改该容器。
2. 需要删除多个元素时，优先使用倒序索引删除或 Qt 提供的删除接口。
3. 发送胜利或失败信号前，先设置结束锁。
4. 一局游戏的结束信号只能发送一次。
5. 页面关闭前停止属于该页面的计时器。
6. 删除被其他对象引用的对象前，先清理引用它的对象。
7. 修改关卡切换逻辑后，至少检查普通胜利、最终胜利和失败三条路径。
8. Debug 版本出现 Qt 断言时，先保留转储文件和完整调用栈，不只根据弹窗中的 Qt 文件名判断原因。

- [Implicit Sharing](https://doc.qt.io/qt-6/implicit-sharing.html)：说明 Qt 的写时复制和数据分离机制。
- [QTimer](https://doc.qt.io/qt-6/qtimer.html)：说明单次计时器、计时器停止和带上下文的 `singleShot()`。
- [Object Trees & Ownership](https://doc.qt.io/qt-6/objecttrees.html)：说明 `QObject` 父子对象的所有权和自动销毁规则。
