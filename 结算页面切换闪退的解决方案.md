# Qt Widgets 游戏结算页面切换闪退的解决方案

鱼浩琳 写于2026/07/13

## 问题描述

在《最后的要塞》中，一局游戏结束后需要从 `GamePage` 切换到 `ResultPage`，玩家可以在结果页面选择：

- 返回关卡选择；
- 重新挑战当前关卡；
- 进入下一关。

实际测试时，无论游戏失败还是过关，都可能在显示结算页面或点击“下一关”后闪退。

这个问题表面上发生在页面切换阶段，因此前几次主要围绕 `MainWindow`、`QStackedWidget` 和结果页面进行修改。但是继续检查后发现，页面切换只是触发问题的时间点，真正的原因是旧战斗场景中仍然存在延迟回调和悬空指针。

---

## 页面切换的基本结构

当前项目使用 `QStackedWidget` 管理页面：

```text
MainWindow
└── QStackedWidget
    ├── StartPage
    ├── LevelSelectPage
    ├── SettingsPage
    ├── StoryPage
    ├── GamePage
    └── ResultPage
```

所有页面在程序启动时创建，再通过下面的方法切换当前可见页面：

```cpp
void MainWindow::switchTo(QWidget *page)
{
    if (!page || m_currentPage == page) {
        return;
    }

    if (auto oldGamePage = qobject_cast<GamePage *>(m_currentPage)) {
        oldGamePage->onPageHiddenByStack();
    }

    m_stack->setCurrentWidget(page);
    m_currentPage = page;

    if (auto newGamePage = qobject_cast<GamePage *>(page)) {
        newGamePage->onPageShownByStack();
    }
}
```

Qt 官方文档说明，`QStackedWidget` 在同一时间只显示一个页面，`setCurrentWidget()` 用于指定当前页面。但是切换页面并不等于销毁页面，也不会自动停止页面中的 `QTimer` 或游戏对象。

[QStackedWidget 官方文档](https://doc.qt.io/qt-6/qstackedwidget.html)

因此，游戏页面隐藏时仍然需要主动暂停或清理战斗逻辑。

## 第一次尝试：直接切换到结果页面

最初的处理方式是在收到游戏结束信号后，立刻停止游戏并切换页面：

```cpp
void MainWindow::showResultPage(bool win, int score)
{
    m_gamePage->stopGame();
    m_resultPage->setResult(win, score, m_lastLevelId, backgroundPath);
    switchTo(m_resultPage);
}
```

调用流程大致如下：

```text
GameController::updateFrame()
  → emit gameFinished(...)
  → GamePage::handleGameFinished(...)
  → emit GamePage::gameFinished(...)
  → MainWindow::showResultPage(...)
  → GamePage::stopGame()
  → 清理正在被 updateFrame() 使用的战斗对象
```

问题在于，`gameFinished` 信号本身可能是在 `updateFrame()`、碰撞检测或敌人移动过程中发出的。此时函数调用栈还没有退出，如果同步执行 `clearGame()`，就可能删除当前帧仍在使用的敌人、子弹或城堡。

因此，直接切换和直接清场存在重入风险。

---

## 第二次尝试：把结算切换放到下一次事件循环

为了避免在战斗帧内部同步清场，第二次改为使用 `QTimer::singleShot(0, ...)`：

```cpp
void MainWindow::showResultPage(bool win, int score)
{
    const int finishedLevelId = m_lastLevelId;

    QTimer::singleShot(0, this, [this, win, score, finishedLevelId]() {
        m_gamePage->stopGame();

        const LevelData levelData = LevelManager::createLevel(finishedLevelId);
        m_resultPage->setResult(
            win,
            score,
            finishedLevelId,
            levelData.backgroundPath);

        switchTo(m_resultPage);
    });
}
```

`singleShot(0, ...)` 不会立刻执行 lambda，而是把操作放到事件循环后续处理。这样当前的 `updateFrame()` 和信号调用栈可以先正常返回，再统一清场和切页。

[QTimer 官方文档](https://doc.qt.io/qt-6/qtimer.html)

这个方案解决了“在当前信号调用过程中删除对象”的风险，但测试时仍然会闪退，说明旧战斗中还有其他异步操作没有被一起取消。

## 第三次尝试：暂时取消得分页面

因为闪退总是在结果页面附近出现，曾尝试取消得分页面，在战斗结束后直接返回关卡选择，或者直接进入下一关。

这个方案只能改变闪退发生时显示的页面，不能处理旧战斗对象的生命周期。只要旧场景仍有延迟回调或悬空指针，在进入关卡选择或下一关后仍然可能崩溃。

同时，完全取消中间页面也会造成体验问题：

- 玩家无法确认上一局是成功还是失败；
- “重新挑战”和“下一关”没有稳定的操作入口；
- 游戏画面突然跳转，缺少关卡之间的停顿和反馈。

因此最终仍然保留 `ResultPage`，并使用当前关卡背景作为结果页面背景。

---

## 最终定位一：命中特效延迟回调访问已删除对象

战斗系统会在子弹命中时创建一个 `QGraphicsPixmapItem`，并在 160ms 后删除：

```cpp
QGraphicsPixmapItem *item = m_scene->addPixmap(effectPixmap);

QTimer::singleShot(160, m_scene, [scene = m_scene, item]() {
    if (item->scene() == scene) {
        scene->removeItem(item);
    }
    delete item;
});
```

这里把 `m_scene` 作为定时回调的 context，但 `GameScene` 会跟随 `GamePage` 长期存在，切换关卡时不会销毁。

另一方面，清理游戏时会调用：

```cpp
m_scene->clearSceneSafely();
```

最终会进入 `QGraphicsScene::clear()`。Qt 官方文档明确说明，`clear()` 会从场景中移除并删除所有 item。

[QGraphicsScene::clear() 官方文档](https://doc.qt.io/qt-6/qgraphicsscene.html#clear)

于是会产生下面的顺序：

```text
命中特效 item 被创建
  → 注册 160ms 延迟回调
  → 游戏结束并清理场景
  → QGraphicsScene::clear() 已经删除 item
  → GameScene 仍然存活
  → 160ms 回调继续执行
  → item->scene() 访问已释放内存
  → 再次 delete item
  → 闪退
```

这就是一个典型的释放后访问和重复删除问题。

### 最终修改

命中特效属于本局战斗，因此定时回调应该绑定到本局的 `BattleSystem`，而不是跨关卡存在的 `GameScene`：

```cpp
QTimer::singleShot(160, this, [scene = m_scene, item]() {
    if (item->scene() == scene) {
        scene->removeItem(item);
    }
    delete item;
});
```

这里的 `this` 是当前 `BattleSystem`。

Qt 官方文档说明，带 context 的 `QTimer::singleShot()` 只有在 context 对象到达触发时间时仍然存活，才会调用 functor。因此清理本局战斗并删除 `BattleSystem` 后，尚未执行的命中特效回调会自动失效。

这使异步任务的生命周期和它所操作的数据保持一致：

```text
BattleSystem 存活
  └── 命中特效回调允许执行

BattleSystem 被清理
  └── 尚未执行的命中特效回调自动取消
```

## 最终定位二：其他子弹仍然追踪已死亡敌人

一名敌人可能同时被多座塔攻击，因此会有多颗子弹保存同一个 `Enemy*`：

```cpp
Enemy *m_targetEnemy;
```

原来的帧处理顺序为：

```text
updateAllBullets()
  → runGlobalCollisionCheck()
  → cleanDeadEnemies()
```

当第一颗子弹击杀敌人后，`cleanDeadEnemies()` 会在当前帧结尾删除这个敌人。但是其他仍在飞行的子弹依然保存它的地址。下一帧执行下面的代码时：

```cpp
bool Bullet::hasLiveTarget() const
{
    return m_targetEnemy && !m_targetEnemy->isDead();
}
```

`m_targetEnemy` 虽然不为 `nullptr`，但已经指向被释放的内存，调用 `isDead()` 就可能闪退。

这个问题特别容易在最后一名敌人死亡时出现，所以从玩家角度看，很像是“进入结算页面导致闪退”。

### 最终修改

为 `Bullet` 增加只读目标接口：

```cpp
Enemy *Bullet::targetEnemy() const
{
    return m_targetEnemy;
}
```

删除敌人前，先删除所有仍然指向它的子弹：

```cpp
for (int i = m_bulletContainer.size() - 1; i >= 0; --i) {
    Bullet *bullet = m_bulletContainer[i];
    if (bullet && bullet->targetEnemy() == enemy) {
        removeBulletAt(i);
    }
}

delete enemy;
```

这样不会再有子弹保存已经失效的敌人地址。

Qt 的对象模型提供 `QPointer` 来保护 `QObject` 指针，对象销毁后会自动变成空指针。但是本项目中的 `Enemy` 继承自 `QGraphicsPixmapItem`，不是 `QObject`，所以当前采用“删除目标前先清理引用者”的方法更适合现有结构。

[Qt Object Model 官方文档](https://doc.qt.io/qt-6/object.html)

## 最终处理：统一关卡入口并阻止页面切换重入

除了修复两个悬空指针，`MainWindow` 还将故事页继续、重新挑战和下一关统一为同一个入口：

```cpp
void MainWindow::queueStartLevel(int levelId)
{
    if (m_levelTransitionPending) {
        return;
    }

    m_levelTransitionPending = true;

    QTimer::singleShot(0, this, [this, levelId]() {
        startLevel(levelId);
        m_levelTransitionPending = false;
    });
}
```

三个入口都调用：

```cpp
queueStartLevel(levelId);
```

`m_levelTransitionPending` 用来阻止按钮连续点击或重复信号造成切换重入。`singleShot(0, ...)` 则保证当前按钮事件处理完成后，再清理旧关卡并加载新关卡。

结果页面的显示也使用相同的保护：

```cpp
if (m_levelTransitionPending) {
    return;
}

m_levelTransitionPending = true;
```

于是最终流程变为：

```text
游戏结束
  → 停止主循环计时器
  → 当前战斗帧返回
  → 删除 BattleSystem，取消本局延迟特效回调
  → 删除指向死亡敌人的剩余子弹
  → 清理 GameScene
  → 设置 ResultPage 的文字和关卡背景
  → QStackedWidget 切换到 ResultPage

玩家点击下一关
  → 切换锁阻止重复点击
  → 当前按钮事件返回
  → 清理旧关卡
  → 加载下一关
  → 切换到 GamePage
```

## 为什么最终方案能够解决问题

前几次修改主要处理“什么时候切换页面”，最终方案同时处理了三个层面：

1. **页面层**：`MainWindow` 统一管理页面和关卡入口；
2. **事件循环层**：使用 `singleShot(0, ...)` 避免在当前信号和战斗帧中同步清场；
3. **对象生命周期层**：异步特效绑定本局 `BattleSystem`，敌人删除前清理所有子弹引用。

其中第三点才是这次闪退问题的真正根源。如果只修改结果页面，旧战斗对象仍然可能在页面切换后访问已经释放的内存。

## 总结

这次问题说明，Qt 游戏中的“页面切换闪退”不一定由页面控件本身造成。`QStackedWidget` 只负责显示哪一个页面，真正需要额外管理的是旧页面背后的计时器、场景对象、延迟回调和裸指针。

解决这类问题时，可以按照下面的顺序检查：

```text
先确认是否在信号调用栈中同步删除对象
  → 再检查所有 QTimer 和延迟回调的 context
  → 再检查 QGraphicsScene::clear() 会删除哪些 item
  → 最后检查是否还有其他对象保存这些 item 的裸指针
```

只有页面切换时序和战斗对象生命周期同时正确，结算页面、重新挑战和下一关流程才能稳定运行。
